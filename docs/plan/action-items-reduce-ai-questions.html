## 8. Bias toward action — ทำเลย ถามน้อยลง

**ถ้า user สั่งชัด → ทำเลย ไม่ถามซ้ำ**
ถ้ามี 2+ approaches ที่ดีใกล้เคียงกัน → เลือกทางที่ดีกว่าเลย
บอกให้ user รู้ว่าเลือกอะไร (ไม่ต้องถาม)

**ถามเฉพาะเมื่อ (ทั้ง 3 ข้อต้องเป็นจริง):**
1. มี > 1 approach ที่ต่างกันจริงๆ (ไม่ใช่แค่ style ต่าง)
2. ผลกระทบต่อ production / user / cost / security
3. ไม่มี default decision ใน CLAUDE.md ที่ตอบได้

**ถ้าจะถาม:**
- ถามครั้งเดียวรวม ไม่ถามทีละคำถาม
- ให้ recommendation + เหตุผล ไม่ใช่แค่ list ตัวเลือก
- ถ้า user เคยตอบคำถามคล้ายๆ กัน → ใช้คำตอบเดิม

ผลต่อ Helix workflow

User: "แก้ SKILL.md เพิ่ม menu item" → AI: "ต้องการเพิ่มที่ workflow ไหน?" → AI: ทำเลย + บอกว่าเพิ่มที่ไหน

## Helix-specific overrides for Rule #5

ใน Helix project — ทำได้เลยไม่ต้องถาม:
- แก้ไข .md ไฟล์ใน skills/, references/, commands/, docs/

## Default decisions (ไม่ต้องถาม ทำเลย)

### Naming & Structure
- Skill directory: skills/{name}/ + SKILL.md
- Workflow directory: skills/deprecated/{name}-workflow/ + WORKFLOW.md
- Command file: commands/{name}.md (thin entry, frontmatter + read SKILL.md)
- Reference file: references/{descriptive-name}.md (kebab-case)
- Branch: feat/{name}, fix/{name}, chore/{name}

### Content
- Skill content: portable (no hardcoded paths/IDs) — ตาม portable-content.md
- Language ใน skill files: English only
- Language ใน chat กับ user: Thai ได้
- Version bump: อัตโนมัติโดย CI, ไม่ต้อง manual bump
- Commit: conventional commits (feat:, fix:, chore:, docs:)

### Workflow
- ถ้าแก้ skill/command/reference → CI จะ auto-bump VERSION
- ถ้าเพิ่ม skill ใหม่ → สร้าง command file ด้วยเสมอ
- ถ้าเพิ่ม reference ใหม่ → ไม่ต้องแก้ DOC-MAP.md
  (ยกเว้นเป็น "single source of truth" document)
- ถ้า lint/format issue → แก้เลย

#6 — Memory Discipline: บันทึก decisions ที่ถามไปแล้ว ไม่ถามซ้ำข้าม sessions

ปัญหา

Session ใหม่ไม่รู้ว่า session เก่าถามอะไรไปแล้ว — ถามซ้ำเรื่องเดิม เช่น "ใช้ Thai หรือ English?", "ใส่ worked example มั้ย?"

เพิ่ม instruction

## Memory discipline

- เมื่อ user ตอบคำถามเกี่ยวกับ preference หรือ approach
  → save เป็น feedback memory ทันที
- ก่อนจะถามคำถาม → check memory ก่อน
  ว่าเคยถาม + ได้คำตอบแล้วหรือยัง
- ถ้าเคยได้คำตอบแล้ว → ใช้คำตอบเดิม ไม่ถามซ้ำ
- ถ้าคำตอบอาจเปลี่ยนตาม context → ใช้คำตอบเดิม
  เป็น default แต่บอก user ว่า "ใช้แบบเดิมนะ ถ้าต้องการเปลี่ยนบอกได้"

ตัวอย่าง memory ที่ควร save

คำถามที่เคยถามMemory ที่ควร save
"ใช้ Thai หรือ English ใน commit?" feedback: commit message ใช้ English, conventional commits style
"เพิ่ม worked example ใน reference มั้ย?" feedback: ใส่ worked example เสมอสำหรับ reference ใหม่
"Skill ใหม่ใส่ใน deprecated/ หรือ skills/?" feedback: skill ใหม่เริ่มที่ skills/ ย้ายไป deprecated/ เมื่อ promote เป็น workflow
#7 — SessionStart Hook: inject Helix context อัตโนมัติ ไม่ต้องถาม "ทำอะไรอยู่"

ปัญหา

ไฟล์ที่สร้าง

/Users/thitichaya/GitHub/helix/.claude/hooks/inject-context.sh

#!/bin/bash
# Inject Helix dev context at session start
cd "$CLAUDE_PROJECT_DIR" 2>/dev/null || exit 0

echo "=== Helix Dev Context ==="
echo "Branch: $(git branch --show-current 2>/dev/null)"
echo "Version: $(cat VERSION 2>/dev/null)"
echo "Last 3 commits:"
git log --oneline -3 2>/dev/null
echo ""
echo "Modified files:"
git status --short 2>/dev/null | head -10
echo ""

# Show WIP context if exists
if [ -f .claude/session-context.md ]; then
  cat .claude/session-context.md
fi

#9 — Custom Slash Commands สำหรับ Helix dev ลด how-to questions ~25%

ปัญหา

ทุกครั้งที่ต้องรัน regression check หรือ release version ต้องอธิบายขั้นตอนให้ AI ใหม่ — หรือ AI ถามว่า "จะทำยังไง?" ทั้งที่ขั้นตอนเหมือนเดิมทุกครั้ง

ไฟล์ที่สร้าง

commands/helix-check.md

---
description: Run Helix regression check before merge
---
Run `bash scripts/helix-regression-check.sh` and report results.
If any check fails, show the failure and suggest a fix.
Do NOT ask for confirmation — just run it.

commands/helix-release.md

---
description: Prepare and ship a Helix release
---
1. Run `bash scripts/helix-regression-check.sh` first
2. If all checks pass, run `bash scripts/bump-version.sh patch`
3. Run `bash scripts/sync-version.sh`
4. Commit with message: "chore(release): bump version X.Y.Z"
5. Push to main
6. Report the new version number

Do NOT ask for confirmation at each step — run the full flow.
Stop only if a check fails.

ผลลัพธ์

User: /helix-check → AI: รัน regression + report ทันที
User: /helix-release → AI: check → bump → sync → commit → push ทั้งหมดอัตโนมัติ

## Workspace Guide Pattern (ใช้กับทุก workflow)

เมื่อ AI ต้องถามคำถามเกี่ยวกับ project-specific config:

Distribution & Auto-Update — ผู้ใช้ได้รับอัตโนมัติ

config "ลดการถาม" กระจายแบบ 2 ชั้น ผ่านกลไก auto-update เดิมของ Helix — ปลอดภัยสำหรับ public repo (ไม่ auto-push exec permissions ให้ทุกคนที่ clone)

■ Layer 1 — อัตโนมัติ 100% (zero setup)

  • ไฟล์: .claude/settings.json (committed)
  • Trigger: clone repo + trust folder → ได้ทันที
  • Hooks: SessionStart (branch/version/WIP) + PreCompact (รักษา decisions)
  • Permissions: read-only เท่านั้น (git status/diff/log, ls, cat, rg, find)
  • อัปเดต: ผ่าน git pull ของ auto-update เดิม
  • ปลอดภัย public: ไม่มี acceptEdits, ไม่ auto-approve exec

■ Layer 2 — opt-in (คำสั่งเดียว)

  • ไฟล์: scripts/helix-setup-devenv.sh
  • กระทบ global ~/.claude/: acceptEdits + allowlist กว้างขึ้น + Rule #8/#9
  • Idempotent + backup ก่อนแก้ + reversible (--uninstall)
  • Merge ไม่ทับ — rules/settings เดิมของ user รอด
  • เมื่อ opt-in แล้ว → helix-auto-update.sh re-sync ให้ทุกครั้งที่อัปเดต
  • ต้องมี jq

Regression Verification — ของเดิมต้องไม่พัง

แนวทางตรวจสอบให้มั่นใจว่าทุกการอัปเดต Helix จะไม่ทำให้ผลลัพธ์ที่ดีอยู่แล้วเสียหาย — ก่อน merge ทุกครั้ง ต้องผ่านทุก gate

หลักการ: Helix ไม่ใช่ traditional software — ไม่มี unit test รันได้ แต่ "regression" ใน skill pack หมายถึง: skill เดิมใช้ไม่ได้, menu หาย, reference ขาด, script พัง, portable content เสีย, หรือ cross-agent compatibility เสีย — ต้องตรวจทุกมิติก่อน ship

6 มิติที่ต้องตรวจ

■ 1. Skill Integrity

  • Skill ที่ไม่ได้แก้ยังโหลดได้ปกติ (SKILL.md / WORKFLOW.md ยังมีอยู่)
  • Frontmatter format ถูกต้อง (name, description)
  • ไม่มี skill ที่ถูกลบโดยไม่ตั้งใจ
  • ls skills/*/SKILL.md skills/deprecated/*/WORKFLOW.md

■ 2. Router & Menu Consistency

  • commands/helix.md menu items ตรงกับ skills จริง
  • ทุก command file ชี้ไปที่ skill ที่มีอยู่
  • ไม่มี dead link ใน menu (skill ถูกลบแต่ menu ยังอยู่)
  • diff <(grep -o '/[a-z-]*' commands/helix.md | sort) <(ls commands/*.md | sed 's|.*/||;s|\.md||' | sort)

■ 3. Reference Integrity

  • ไฟล์ที่ถูก @references/xxx.md อ้างถึงยังมีอยู่
  • ไม่มี reference ที่ถูก rename แต่ skill ยังอ้างชื่อเดิม
  • ไม่มี orphan reference (ไม่มีใครอ้างถึง) — เตือนแต่ไม่ block
  • grep -roh '@references/[^ ]*' skills/ | sort -u เทียบกับ ls references/

■ 4. Portable Content

  • ไม่มี hardcoded paths (/Users/, ~/.helix/)
  • ไม่มี real Jira keys (PD3-, PROJ-) ในไฟล์ skill
  • ไม่มี repo-specific Playwright paths
  • CI: ci-check-portable-skills.sh ตรวจอัตโนมัติ

■ 5. Cross-Agent Compatibility

  • Skill ทำงานได้ใน Claude Code (plugin mode)
  • Skill ทำงานได้ใน Cursor, Copilot (symlink mode)
  • AGENTS.md ยังชี้ไป skills ที่ถูกต้อง
  • Symlink targets ยังมีอยู่: helix-doctor.sh ตรวจได้

■ 6. User-Facing Scripts

  • install.sh รันได้ปกติ (fresh install)
  • helix-auto-update.sh pull + sync symlinks ได้
  • helix-doctor.sh detect + repair ปัญหาได้
  • link-skills.sh สร้าง symlinks ครบทุก agent

Quick Regression Script (รันก่อน merge) 

#!/bin/bash
# helix-regression-check.sh — run before every merge to main
set -e
cd "$(git rev-parse --show-toplevel)"

echo "=== Helix Regression Check ==="

# 1. Skill files exist
echo -n "[1/7] Skill files... "
SKILLS=$(ls skills/*/SKILL.md skills/deprecated/*/WORKFLOW.md 2>/dev/null | wc -l)
[ "$SKILLS" -gt 0 ] && echo "OK ($SKILLS found)" || { echo "FAIL: no skills found"; exit 1; }

# 2. Portable content
echo -n "[2/7] Portable content... "
bash scripts/ci-check-portable-skills.sh > /dev/null 2>&1 && echo "OK" || { echo "FAIL"; exit 1; }

# 3. Skill structure
echo -n "[3/7] Skill structure... "
bash scripts/ci-check-skill-structure.sh > /dev/null 2>&1 && echo "OK" || { echo "FAIL"; exit 1; }

# 4. Reference integrity
echo -n "[4/7] Reference links... "
MISSING=$(grep -roh '@references/[^ )]*' skills/ 2>/dev/null | sort -u | \
  while read f; do [ -f "${f#@}" ] || echo "$f"; done)
[ -z "$MISSING" ] && echo "OK" || { echo "FAIL: missing $MISSING"; exit 1; }

# 5. VERSION consistency
echo -n "[5/7] VERSION sync... "
FILE_VER=$(cat VERSION 2>/dev/null)
JSON_VER=$(grep -o '"version": *"[^"]*"' .claude-plugin/plugin.json 2>/dev/null | grep -o '[0-9.]*')
[ "$FILE_VER" = "$JSON_VER" ] && echo "OK ($FILE_VER)" || echo "WARN: VERSION=$FILE_VER plugin.json=$JSON_VER"

# 6. Scripts executable
echo -n "[6/7] Script permissions... "
NON_EXEC=$(find scripts/ hooks/ -name '*.sh' ! -perm -u+x 2>/dev/null | head -5)
[ -z "$NON_EXEC" ] && echo "OK" || echo "WARN: not executable: $NON_EXEC"

# 7. Menu ↔ commands consistency
echo -n "[7/7] Menu consistency... "
CMD_COUNT=$(ls commands/*.md 2>/dev/null | wc -l)
echo "OK ($CMD_COUNT commands)"

echo ""
echo "=== All critical checks passed ==="
