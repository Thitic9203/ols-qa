Test.md Format — Agent-Native Test Case Output

News #2: TestMu Test.md Effort: Low Impact: High

Why

Helix สร้าง test case output เป็น Markdown table อยู่แล้ว (tc-fe-prep, tc-api-prep) — TestMu เพิ่งประกาศ Test.md ซึ่งเป็น structured Markdown format ที่ทั้งคนอ่านง่ายและ AI agent รันได้โดยตรง ถ้า Helix adopt format นี้ output ของ tc-prep workflows จะ reusable มากขึ้น — agent ตัวอื่น (Kane CLI, Playwright agent) หยิบไปรันได้เลยโดยไม่ต้อง parse

Mapping กับ Helix

Helix Component Action File
tc-fe-prep output format เพิ่ม Test.md export option ข้าง CSV/Excel skills/deprecated/tc-fe-prep-workflow/
tc-api-prep output format เพิ่ม Test.md export option ข้าง CSV/Excel skills/deprecated/tc-api-prep-workflow/
CSV export rules เพิ่ม Test.md format spec ใน reference references/csv-export-rules.md
TC quality standards Map ISTQB attributes → Test.md fields references/tc-quality-standards.md

Implementation Steps

  1. ศึกษา Test.md spec จาก TestMu AI — ดู field structure, metadata format ดูว่า Test.md กำหนด test step, expected result, precondition ยังไง
  2. สร้าง reference ใหม่ references/test-md-format.md — mapping ISTQB fields → Test.md fields เป็น single source of truth สำหรับ format spec
  3. เพิ่ม delivery option "Test.md" ใน tc-fe-prep + tc-api-prep workflows อยู่ข้าง CSV / Excel / Jira comment ที่มีอยู่แล้ว
  4. เขียน export script scripts/export-test-md.py แปลง Markdown table → Test.md format ใช้ pattern เดียวกับ export-markdown-table-to-csv.py ที่มีอยู่
  5. เพิ่ม CI check — validate Test.md output ใน regression suite เพิ่มเป็น point #8 ใน helix-regression-check.sh
i Test.md เป็น format ใหม่ (พ.ค. 2026) — spec อาจเปลี่ยน ควร abstract format ไว้ใน reference เดียวเพื่อแก้จุดเดียว
! ต้องไม่ break CSV/Excel export ที่มีอยู่ — Test.md เป็น option เพิ่ม ไม่ใช่ replacement

Self-Healing Selector Reference — ลด Flaky Test ใน Playwright Workflow

News #3: KaneAI Intelligent Retry Effort: Medium Impact: High

Why

Helix มี testing-ticket-workflow ที่ใช้ Playwright รันจริง + playwright-preflight.md check — ปัญหาที่เจอบ่อยคือ selector เปลี่ยนแล้ว test พัง KaneAI เพิ่งเพิ่ม intelligent retry + self-healing selectors — แนวคิดนี้ adapt มาเป็น discipline doc ที่สอน AI agent ให้เลือก resilient selectors ตั้งแต่แรก + มี fallback strategy เมื่อ selector fail

Mapping กับ Helix

Helix Component Action File
Playwright preflight เพิ่ม selector resilience checklist references/playwright-preflight.md
Testing ticket workflow เพิ่ม retry-with-fallback-selector step skills/deprecated/testing-ticket-workflow/
QA debug discipline เพิ่ม flaky selector diagnosis flow references/qa-debug-discipline.md
New reference สร้าง selector strategy guide references/resilient-selectors.md

Implementation Steps

  1. สร้าง references/resilient-selectors.md — selector priority hierarchy data-testid > role+name > text content > CSS class > XPath (ห้าม) + fallback chain pattern
  2. อัปเดต playwright-preflight.md — เพิ่ม "Selector Resilience Gate" ก่อนรัน test ต้อง verify ว่า selectors ใช้ data-testid หรือ role-based
  3. เพิ่ม retry discipline ใน testing-ticket workflow — เมื่อ selector fail ให้ agent ลอง fallback ก่อน fail pattern: try data-testid → try role → try text → report as genuine failure
  4. เพิ่ม flaky test pattern ใน qa-debug-discipline.md แยก "selector เปลี่ยน" vs "UI bug จริง" vs "timing issue"
i ไม่ใช่ self-healing อัตโนมัติแบบ KaneAI (ต้องซื้อ) — แต่เป็น discipline ที่สอน AI agent ให้ทำ self-healing logic เอง = free
! Retry fallback ต้องมี limit (max 3 strategies) — ไม่งั้น agent จะ loop ไม่รู้จบ

Proactive QA Trigger — Auto-detect เมื่อ PR ต้องการ Test Case

News #1: Claude Proactive Workflows News #5: Mabl Active Coverage Effort: Medium Impact: Very High

Why

ตอนนี้ Helix เป็น reactive — QA ต้องสั่ง /helix เอง Anthropic เพิ่ง announce Proactive Workflows ที่ Claude ตรวจจับ condition แล้ว trigger action เอง + Mabl ปล่อย Active Coverage ที่ validate quality ทุกครั้งที่โค้ดเปลี่ยน — Helix ใช้ hook system (SessionStart, PreCompact) อยู่แล้ว สามารถเพิ่ม proactive trigger ที่ตรวจจับว่า "PR นี้แตะ feature X → ควรรัน tc-fe-prep" ได้

Mapping กับ Helix

Helix Component Action File
SessionStart hook เพิ่ม PR context check — detect ว่ามี Jira ticket linked ที่ยังไม่มี TC hooks/session-start
Skill routing เพิ่ม proactive_triggers ใน SKILL.md ให้ match PR/branch pattern references/skill-routing.md
New reference สร้าง proactive QA trigger rules references/proactive-qa-triggers.md
Helix router เพิ่ม suggestion mode — แนะนำ workflow ที่ควรรัน (ไม่ auto-run) skills/helix/SKILL.md

Implementation Steps

  1. สร้าง references/proactive-qa-triggers.md — กฎที่ระบุว่า condition ไหน trigger workflow ไหน เช่น branch ชื่อ feat/* + Jira ticket type=Story → แนะนำ tc-fe-prep; branch ชื่อ fix/* + Jira type=Bug → แนะนำ retest-bug
  2. อัปเดต hooks/session-start — inject git branch + linked ticket context ใช้ inject-context.sh ที่มีอยู่ + เพิ่ม Jira ticket detection
  3. อัปเดต helix router SKILL.md — เพิ่ม proactive suggestion block "Based on your current branch feat/login-redesign and linked JIRA-123, I recommend running tc-fe-prep"
  4. เพิ่ม proactive_triggers ใน SKILL.md frontmatter ของแต่ละ workflow ให้ Claude match pattern อัตโนมัติ ไม่ต้องรอ user เลือกจาก menu
  5. เพิ่ม opt-out mechanism — HELIX_PROACTIVE=0 env var ปิด suggestion ได้ เพื่อไม่ให้ annoying ถ้า QA ต้องการ manual control
! Proactive = suggest only, ไม่ auto-run — ป้องกัน agent ทำงานที่ user ไม่ได้สั่ง (Pluton Rule #5)
i ต้องการ Jira MCP connection — ถ้าไม่มี fallback ใช้ branch name + commit message แทน
! ไม่ใช้ Managed Agents API (ต้อง pay) — ใช้ hook system ของ Claude Code ที่ free อยู่แล้ว

Regression Verification — ของเดิมต้องไม่พัง

แต่ละ initiative เป็น additive (เพิ่มของใหม่ข้างของเดิม) — ก่อน merge ทุกครั้งต้องพิสูจน์ว่า CSV/Excel export เดิม, Playwright workflow เดิม, และ reactive /helix เดิม ยังทำงานครบเหมือนเดิม

หลักการ: Helix ไม่ใช่ traditional software — ไม่มี unit test รันได้ตรงๆ "regression" ใน skill pack หมายถึง: skill เดิมใช้ไม่ได้, menu หาย, reference ขาด, script พัง, portable content เสีย, cross-agent compatibility เสีย — บวกกับ 3 พื้นผิวใหม่จาก plan นี้: export format เดิมพัง, selector fallback loop ไม่จบ, หรือ proactive trigger auto-run โดยไม่ได้สั่ง (ฝ่าฝืน Rule #5)

มิติที่ต้องตรวจ (3 ต่อ initiative + 3 ทั่วไป)

■ #1 Test.md — Export ไม่ถดถอย

  • CSV export เดิมยังออกได้ครบทุก column (UTF-8 + BOM)
  • Excel/Jira comment delivery เดิม ไม่เปลี่ยน behavior
  • Test.md เป็น option เพิ่ม ไม่ใช่ replacement
  • Format spec อยู่ใน references/test-md-format.md จุดเดียว
  • มี ISTQB field ครบเมื่อแปลงไป Test.md (ไม่ตกหล่น)

■ #2 Selectors — Fallback มีขอบเขต

  • playwright-preflight.md เดิมยัง gate ได้ปกติ
  • Fallback chain จำกัด max 3 strategies ไม่ loop ไม่จบ
  • เมื่อครบ 3 แล้ว fail → report เป็น genuine failure
  • testing-ticket workflow เดิมไม่ break (เพิ่ม step ไม่ลบ)
  • แยก "selector เปลี่ยน" vs "UI bug จริง" vs "timing" ได้

■ #3 Proactive — Suggest-Only

  • Proactive = แนะนำเท่านั้น ห้าม auto-run (Rule #5)
  • HELIX_PROACTIVE=0 ปิด suggestion ได้จริง
  • SessionStart hook เดิม (branch/version/WIP) ยัง inject ครบ
  • ไม่มี Jira MCP → fallback ใช้ branch/commit แทน ไม่ error
  • ไม่เรียก Managed Agents API (ไม่มี spend — Rule #3)

■ 4. Skill & Reference Integrity

  • Skill ที่ไม่ได้แก้ยังโหลดได้ (SKILL.md/WORKFLOW.md ครบ)
  • reference ใหม่ 3 ไฟล์มีจริง + ถูกอ้างถูกชื่อ
  • ไม่มี dead @references/ link
  • DOC-MAP.md อัปเดตถ้า reference ใหม่เป็น single source of truth

■ 5. Portable Content

  • export-test-md.py ไม่มี hardcoded path/ID
  • reference ใหม่ไม่มี /Users/, real Jira keys, repo paths
  • CI ci-check-portable-skills.sh ผ่าน

■ 6. Cross-Agent & Scripts

  • Skill ทำงานได้ทั้ง Claude Code + Cursor/Copilot
  • SessionStart hook ใหม่ไม่ break agent ที่ไม่มี hook (Gemini/Codex)
  • export-test-md.py รันได้, มี +x
  • helix-doctor.sh ยัง detect/repair ได้
